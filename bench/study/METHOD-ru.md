# Как это устроено: метрики, замер, PCA, логрег

Спутник к [`README-ru.md`](README-ru.md) — для тех, кто хочет понять, *как* мы всё считали. Без
формул-простыней, но по существу.

## 1. Метрики — что это и как считаются

Всё статически, без запуска кода. sloplint переиспользует **парсер Ruff**: каждый `.py` разбирается в
AST (дерево) и поток токенов, дальше мы ходим по дереву и агрегируем. Каждая метрика считается на
своём уровне — функция, класс, модуль, пакет — и потом сворачивается в средние/максимумы/доли по репо.

**Сложность функции**
- *Цикломатическая* — по сути «число веток + 1»: каждый `if`/`for`/`while`/`and`/`or`/`except`
  добавляет +1. Сколько независимых путей через функцию.
- *Когнитивная* (метод SonarSource) — то же, но с **штрафом за вложенность**: `if` внутри `if`
  внутри цикла «дороже», чем три `if` подряд. Ближе к тому, насколько тяжело это читать человеку.
- Считаем и *среднее* по всем функциям, и *максимум* (хвост — один монстр среди нормальных).

**Размер единиц** — длина функции (число содержательных строк), глубина вложенности, число
параметров; на уровне модуля — его длина (god-модули).

**Классы (метрики CK)** — WMC (сумма сложностей методов), DIT (глубина наследования), NOC (число
наследников), LCOM (несвязность методов), CBO (сцепление с другими классами).

**Дублирование** — ищем near-duplicate функции: нормализуем тело (убираем имена/литералы) и сравниваем
структуру. `clone_ratio` = доля функций, у которых есть близкий двойник.

**Тесты** (без запуска!) — делим файлы на «продакшн» и «тесты» по путям (`tests/`, `test_*.py`).
`test_code_ratio` = строки тестов / строки продакшна. И **субстанция**: `assertion_density` (сколько
`assert` на функцию) и `assertion_free_rate` (доля тест-функций вообще без проверок — «тесты для
галочки»). Это ловит театр тестирования, которого по числу файлов не видно.

**Документация / типы** — доля функций с докстрингом и доля параметров с type-хинтами.

**Архитектура (пакеты)** — строим граф импортов между модулями и считаем: `propagation_cost`
(насколько изменение одного модуля расходится по системе — плотность транзитивного замыкания),
циклы/«тангелы» (взаимные импорты, через алгоритм Тарьяна для SCC), модулярность (насколько граф
бьётся на сообщества — Louvain), и долю недекомпозированного кода на верхнем уровне модуля.

**Как это собиралось на 10k.** Для каждого из ~12 000 отобранных репозиториев: shallow-клон → один
прогон `sloplint metrics --format json` → разворачиваем ~118 чисел в строку таблицы → **удаляем клон**
(чтобы диск не рос). Плюс по GitHub API тянем внешние метки (звёзды, CI, релизы, контрибьюторы, доля
AI-коммитов по трейлерам в сообщениях). На выходе — таблица `репо × (118 метрик + метки)`.

## 2. Как фитили PCA (поиск независимых осей)

Вопрос был: 118 метрик — это сколько *реально разных* измерений? Многие же дублируют друг друга
(средняя когнитивная и средняя цикломатическая почти одно и то же). Это и есть задача для PCA.

Пошагово (скрипт `axes.py`):

1. **Берём 18 содержательных метрик** (и только те репо, где все они посчитаны).
   *Почему именно 18, а не все 118?* Полная панель полна **дублей**: risk-tier счётчики
   (`*_risk.low/moderate/high/very_high`) — это бакеты той же метрики; `p95_*` — ещё одна сводка
   того же распределения; сырые `files`/`functions`/`total_loc` — это размер (его мы выносим); под-
   счётчики вроде `functions_in_clones` — функции от `clone_ratio`. Свалить всё в PCA — значит раздуть
   факторы артефактами кодировки (сложность, закодированная 5 раз, даст ложно-огромную ось). Поэтому —
   **по 1–2 непрерывных представителя на понятие**: сложность (avg/max cog+cyc), размер единиц
   (function_loc, nesting, god_units), дубли (clone), декомпозиция (top-level), доки/комменты/типы,
   архитектура (propagation, cycles, modularity), тесты (ratio, assert-density, assert-free). Это
   судейский выбор — другая курация дала бы чуть другие оси; но цель PCA была — структура *понятий*, а
   не дублей. Коллинеарную пару avg_cog/avg_cyc оставили нарочно — увидеть, как она схлопнется в один
   фактор (так и вышло — F1), было проверкой, что они дублируют друг друга.
2. **Убираем размер.** Мы уже знали, что размер — главный мешающий фактор (большой репо → больше
   всего). Поэтому каждую метрику **регрессим на `log(LOC)` и берём остаток** — то, что *не объясняется
   просто объёмом кода*. Дальше работаем с этими остатками, т.е. смотрим на качество «помимо размера».
3. **Стандартизуем** (z-score: вычитаем среднее, делим на разброс) — чтобы метрики в разных шкалах
   были сравнимы.
4. **Строим матрицу корреляций** 18×18 и **раскладываем её на собственные числа/векторы**
   (eigendecomposition). Каждый собственный вектор — это «новая ось» (комбинация метрик), а собственное
   число — сколько дисперсии эта ось забирает.
5. **Сколько осей оставить** — правило Кайзера: берём оси с собственным числом > 1 (т.е. ось, несущая
   больше, чем одна метрика в среднем). Получилось ~7. Важно: это завышающий критерий — две последние
   оси (1.05 и 1.03) еле прошли порог и оказались слабым «мусором».
6. **Varimax-ротация** — поворачиваем оставшиеся оси так, чтобы каждая метрика грузилась в основном на
   *одну* ось. Без этого оси «размазаны» и нечитаемы; после — каждая ось получает понятное имя по тому,
   какие метрики на ней висят (loadings).

Итог: ~6 крепких независимых осей (сложность, хвост, дубли, тесты, доки/типы, архитектура). Главный
вывод PCA: **ни одна ось не доминирует** (первая — лишь 17% дисперсии), значит сжать всё в один балл
нельзя без потери смысла → бейдж = профиль из осей, а не одно число.

## 3. Как зафитили логрег (а отделяет ли AI?)

Дальше — отдельный вопрос: если у нас есть эти оси, **можно ли по ним предсказать, AI-репо или нет?**
Это классическая задача классификации → логистическая регрессия (`ai_classify.py`).

1. **Признаки** — позиция репо на каждой из 7 осей (factor scores). Они уже size-controlled (см. п.2),
   то есть мы спрашиваем про *структуру качества*, а не «AI-репы просто крупнее».
2. **Метка** — `1` если ≥50% последних коммитов с AI-трейлером, `0` если совсем без. Спорную середину
   выбросили.
3. **Фит** — логрег предсказывает вероятность «это AI» как взвешенную сумму осей (веса подбираются
   градиентным спуском, с L2-регуляризацией чтобы не переобучиться).
4. **Честная проверка — 5-fold кросс-валидация.** Делим репо на 5 частей, учимся на 4, проверяем на
   5-й, повторяем 5 раз. Так качество меряется **на невиданных данных**, без самообмана.
5. **Метрика качества — AUROC**: вероятность, что случайный AI-репо получит балл выше случайного
   не-AI. 0.5 = монетка, 1.0 = идеально.

Что вышло:
- только оси (без размера) → **AUROC 0.74**;
- только размер `log(LOC)` (бейзлайн) → 0.69;
- оси + размер → 0.79.

То есть оси несут *сигнал помимо размера* (бьют бейзлайн). Но 0.74 — это «средне», и по **весам** видно,
*чем* отделяется AI: сильнее всего грузят `docs/typing` (+0.87) и средняя сложность (+0.63), а «хвост»
не помогает вовсе. Перевод: классификатор ловит AI по **стилю** (много докстрингов/типов), а не по
плохости. Поэтому строить правило «похоже на AI → плохо» нельзя — оно флагует *хорошо документированный*
код. Это и есть главный смысловой результат: **slop — это про плохость, а не про происхождение.**
